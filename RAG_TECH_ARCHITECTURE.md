## 1. RAG 系统概述

### 1.1 什么是 RAG？

**RAG（Retrieval-Augmented Generation）** 即检索增强生成，是一种结合了信息检索与大型语言模型生成的技术架构。

```
┌─────────────────────────────────────────────────────────────┐
│                         RAG 流程图                           │
└─────────────────────────────────────────────────────────────┘

     用户提问
        │
        ▼
┌─────────────────┐
│   Query理解      │ ← 意图识别、问题分类、关键词提取
└─────────────────┘
        │
        ▼
┌─────────────────┐
│   检索引擎       │ ← 向量检索 / 关键词检索 / 混合检索
└─────────────────┘
        │
        ▼
┌─────────────────┐
│   上下文构建     │ ← Prompt 组装、上下文选择
└─────────────────┘
        │
        ▼
┌─────────────────┐
│   LLM 生成       │ ← 基于上下文 + 知识回答
└─────────────────┘
        │
        ▼
     最终回答
```

### 1.2 RAG vs 传统 LLM

| 对比维度 | 纯 LLM | RAG 系统 |
|---------|-------|---------|
| **知识来源** | 训练数据 | 外部知识库 + 模型知识 |
| **实时性** | 依赖训练数据日期 | 可实时更新知识库 |
| **幻觉问题** | 较高 | 显著降低 |
| **可解释性** | 低 | 高（可溯源到原文） |
| **部署成本** | 高（需微调） | 中（仅需检索系统） |
| **隐私安全** | 差（数据用于训练） | 好（数据本地存储） |

### 4.2 文本分块策略对比

| 分块策略 | 原理 | 优点 | 缺点 | 适用场景 |
|---------|-----|-----|-----|---------|
| **固定长度** | 按字符/词数硬切 | 简单快速 | 可能切断句子 | 通用场景 |
| **滑动窗口** | 重叠滑动切分 | 保留上下文 | 冗余增加 | 问答系统 |
| **语义分段** | 按主题/段落切 | 语义完整 | 计算成本高 | 文档分析 |
| **递归分块** | 递归尝试细分 | 适应性最强 | 复杂度高 | 混合文档 |
| **特殊分隔符** | 按 Markdown/HTML 切 | 保持结构 | 依赖格式 | 技术文档 |

```
滑动窗口分块示意 (chunk_size=10, overlap=3):

原始文本: "ABCDEFGHIJKLMNOPQRSTUVWXYZ"
           0         1         2         3
           0123456789012345678901234567890

Chunk 0:  "ABCDEFGHIJ"        (0-10)
Chunk 1:  "HIJKLMNOP"         (7-16, 重叠 HIJ)
Chunk 2:  "NOPQRSTUV"         (13-22, 重叠 NOP)
Chunk 3:  "STUVWXYZ"          (19-28, 重叠 STU)
```

### 4.3 本项目分块实现

```python
def _split_text(self, text: str) -> List[str]:
    """带重叠的滑动窗口分块"""
    if len(text) <= self.chunk_size:
        return [text]

    chunks = []
    start = 0
    while start < len(text):
        end = min(start + self.chunk_size, len(text))
        chunk = text[start:end].strip()
        if chunk:
            chunks.append(chunk)

        if end >= len(text):
            break
        start = max(end - self.chunk_overlap, start + 1)
    return chunks
```

### 6.2 检索类型对比

| 检索类型 | 原理 | 优点 | 缺点 | 适用场景 |
|---------|-----|-----|-----|---------|
| **稀疏检索 (BM25/TF-IDF)** | 词频统计 | 可解释、关键词友好 | 无法捕捉语义 | 专有名词查询 |
| **稠密检索 (Dense)** | 向量嵌入 | 语义理解强 | 需要向量模型 | 语义相似查询 |
| **混合检索 (Hybrid)** | 稀疏+稠密融合 | 兼顾两者优点 | 复杂度高 | 通用场景 |
| **重排序 (Rerank)** | Cross-Encoder 二阶段 | 精度最高 | 计算成本高 | 高精度场景 |

### 6.3 本项目检索实现 (TF-IDF)

```python
def _score_document(self, query: str, query_terms: list[str], text: str) -> float:
    """TF-IDF 评分逻辑"""
    score = 0.0

    # 1. 完整查询匹配（最高权重）
    if normalized_query in normalized_text:
        score += min(len(normalized_query), 24) * 3.0

    # 2. 各术语匹配（多次出现得分更高）
    for term in query_terms:
        occurrences = normalized_text.count(term)
        if occurrences > 0:
            score += occurrences * max(len(term), 1)

    return score
```

### 7.3 本项目 LLM 集成

```python
class QwenChat:
    def __init__(self, api_key: str, model: str, base_url: str) -> None:
        self.client = OpenAI(api_key=api_key, base_url=base_url)
        self.model = model

    def answer(self, question: str, contexts: List[dict]) -> str:
        # 构建上下文文本
        context_text = "\n\n".join([
            f"[文件: {item['path']} | 分片: {item['chunk_id']} | 相似度: {item['score']:.4f}]\n{item['text']}"
            for item in contexts
        ])

        messages = [
            {
                "role": "system",
                "content": (
                    "你是一个本地知识库问答助手。"
                    "只能优先依据提供的检索上下文回答。"
                    "如果上下文不足，请明确说明'根据当前知识库内容无法确定'，不要编造事实。"
                ),
            },
            {
                "role": "user",
                "content": (
                    f"用户问题：{question}\n\n"
                    f"检索上下文：\n{context_text}\n\n"
                    "请基于以上内容给出中文回答，并在结尾列出引用到的文件路径。"
                ),
            },
        ]

        response = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            temperature=0.2,  # 低温度保证稳定性
        )
        return response.choices[0].message.content or "模型没有返回内容。"
```

### 7.4 Prompt 优化策略

| 策略 | 描述 | 效果提升 |
|-----|------|---------|
| **Few-shot 示例** | 提供问答示例 | 10-30% |
| **Chain-of-Thought** | 让模型逐步推理 | 复杂问题显著 |
| **上下文压缩** | 减少无关信息 | 减少幻觉 |
| **引用标注** | 要求模型引用来源 | 提高可信度 |
| **温度控制** | 控制随机性 (0.1-0.3) | 稳定性 |

### 9.2 向量数据库对比

### C. 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|-----|---------|---------|
| PDF 无法解析 | 缺少 PyPDF2/pdfplumber | `pip install PyPDF2 pdfplumber` |
| 检索无结果 | 编码问题 / 分块太小 | 检查文件编码、调整 chunk_size |
| LLM 超时 | API 限流 / 网络问题 | 添加重试、使用国内 API |
| 内存溢出 | 向量/文档太大 | 分批处理、减小 chunk_size |
| 索引加载慢 | 文件太大 | 使用 SSD、增量加载 |
